# Fonctionnement du contexte IA

Ce projet a été initialisé par le <span style="color:#3b82f6;">Wizard Living-Documentation</span>.

Le wizard a généré pour vous des fichiers d'entrée IA dans le dossier parent du dossier de documentation :

```text
AGENTS.md
CLAUDE.md
memory/MEMORY.md
```

Ces fichiers contiennent un ensemble de principes et de règles permettant à votre agent IA (**Claude ou Codex principalement**) de pouvoir utiliser l'outil **Living Documentation** de manière appropriée et ainsi de vous fournir un outil de suivi et d'interaction avec les LLM et les tâches que ces derniers réalisent pour vous.

Les fichiers suivants sont des liens symboliques vous permettant de pouvoir les consulter au sein de l'outil Living Documentation tout en permettant à vos agents de nativement les utiliser :

```text
<docs-folder>/AI/AGENTS.md -> <project-root>/AGENTS.md
<docs-folder>/AI/CLAUDE.md -> <project-root>/CLAUDE.md
<docs-folder>/AI/MEMORY.md -> <project-root>/memory/MEMORY.md
```

Autrement dit, les liens ne pointent pas vers un autre fichier dans le dossier de documentation : ils exposent dans `AI/` les fichiers réels utilisés par les agents à la racine du projet.

## Rôle des fichiers

`AGENTS.md` est le point d'entrée pour Codex et les outils de type agent.

`CLAUDE.md` est le point d'entrée pour Claude.

`memory/MEMORY.md` est l'index de mémoire locale du projet.

`AI/PROJECT-INSTRUCTIONS.md` est le guide de travail partagé par tous les assistants IA du projet.

`AI/PROJECT-STACK.md` décrit la stack, l'arborescence source utile, les concepts centraux et les conventions structurantes du projet.

`AI/PROJECT-USEFUL-COMMANDS.md` liste les commandes réellement utiles pour installer, lancer, tester, vérifier et maintenir le projet.

`AI/rules/*.md` contient les règles réutilisables que les assistants IA doivent appliquer lorsqu'ils effectuent des modifications sur le projet.

`ADRS/*.md` contient les décisions durables. Les ADR (Architecture Decision Records) expliquent pourquoi le projet fonctionne d'une certaine manière, ils sont automatiquement créés par l'agent IA lorsqu'il réalise ses tâches tel que cela lui est expliqué dans les instructions projet.

## Vérification après le premier démarrage

Ouvrir la page **Contexte IA** depuis la barre principale.

Cette page doit vous permettre de vérifier :

- que `AGENTS.md`, `CLAUDE.md` et `memory/MEMORY.md` apparaissent comme documents dans `AI/` ;
- que `PROJECT-INSTRUCTIONS.md`, `PROJECT-STACK.md` et `PROJECT-USEFUL-COMMANDS.md` sont visibles et adaptés à votre projet ;
- que les règles IA sont visibles dans `AI/rules/` ;
- que l'explorateur MCP affiche le serveur `living-documentation`, l'endpoint `/mcp`, les outils disponibles et les prompts.

Si l'explorateur MCP n'affiche pas les outils, l'assistant IA ne pourra pas utiliser Living Documentation comme source de vérité active. Il pourra encore proposer des changements de documentation, mais il devra signaler que les métadonnées et les hashes n'ont pas pu être mis à jour via MCP.

## Workflow IA attendu

Pour information, le fonctionnement d'un assistant IA est le suivant :

1. Lire son point d'entrée : `AGENTS.md` ou `CLAUDE.md`.
2. Suivre le lien vers `AI/PROJECT-INSTRUCTIONS.md`.
3. Lire `AI/PROJECT-STACK.md` pour comprendre les choix techniques, l'arborescence utile et les concepts du projet.
4. Lire `AI/PROJECT-USEFUL-COMMANDS.md` pour utiliser les bonnes commandes au lieu de les deviner.
5. Lire `memory/MEMORY.md` et charger seulement les fichiers mémoire pertinents.
6. Lire les règles applicables dans `AI/rules/*.md`.
7. Inspecter le frontmatter des ADR lorsqu'une tâche peut toucher une décision existante.
8. Réaliser la modification demandée.
9. À la fin d'une feature cohérente, mettre à jour la documentation durable si nécessaire en utilisant le MCP Living-Documentation intégré.
10. Mettre à jour `PROJECT-STACK.md` ou `PROJECT-USEFUL-COMMANDS.md` si la tâche rend leur contenu faux, incomplet ou obsolète.
11. Attacher les métadonnées Living Documentation aux documents créés ou mis à jour lorsque des fichiers source prouvent leur contenu.

## ADR

Le starter inclut un ADR exemple :

```text
ADRS/2026_01_01_[ADR]_example_architecture_decision.md
```

L'IA peut l'utiliser comme template lorsqu'une décision significative doit être enregistrée.

Un ADR est pertinent lorsqu'un changement affecte l'architecture, un contrat public, un format de stockage, un workflow, un protocole ou une convention durable. Il n'est pas nécessaire pour les corrections triviales ou les détails d'implémentation évidents dans le code.

## Métadonnées et fiabilité

Living Documentation peut lier un document à des fichiers source. Chaque lien stocke un hash du fichier source afin que la jauge de fiabilité montre si le document a pu dériver du code.

Quand une IA crée ou met à jour un ADR ou un document technique, elle doit attacher les fichiers source qui prouvent ou implémentent le document. Si les outils MCP sont disponibles, préférer `add_metadata` et `refresh_metadata`.

Si les métadonnées ne peuvent pas être mises à jour depuis l'environnement courant, l'IA doit le dire et lister les fichiers à attacher manuellement.

## Nettoyage

Après initialisation, supprimez seulement ce qui n'est pas utile au projet.

Nettoyage courant :

- garder ce document si les futurs contributeurs doivent comprendre la structure initialisée ;
- adapter `PROJECT-STACK.md` et `PROJECT-USEFUL-COMMANDS.md` dès que les premières commandes et conventions réelles du projet sont connues ;
- garder l'ADR exemple seulement si vous voulez un template dans le projet.
